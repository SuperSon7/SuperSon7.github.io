무중단 인프라 마이그레이션
다운타임 97.22% 단축 · 정상 처리율 99.9965%
단일 EC2(앱+DB 동거) → 앱/DB 분리 전환. nginx stream proxy를 앱과 DB 사이에 삽입하여 앱 코드 변경 없이 upstream만 교체. 쓰기 불가 구간 5초, 롤백은 nginx reload 한 번.

설계 판단

  • DNS TTL 전환 배제 — 캐시 잔존 시 라우팅 불확실, 즉시 롤백 불가
  • CDC/DMS/ProxySQL 배제 — 단일 인스턴스에서 과도한 복잡성과 단일 장애점
  • nginx stream 선택 — 가장 낮은 복잡도로 즉시 롤백 가능

실행 순서

  • nginx stream proxy 삽입 (앱 → proxy → 기존 EC2 DB)
  • EC2 DB super_read_only 활성화
  • binlog 동기화 확인 (lag = 0)
  • proxy upstream을 RDS로 교체 (nginx reload)
  • KILL CONNECTION으로 HikariCP 강제 재연결

03
SLO / Error Budget — Google SRE 방법론
SLO 4개 정의 · 이론 vs 실환경 한계 발견·해결
단순 threshold가 아닌 Error Budget Burn Rate 방식 운영. 저트래픽 서비스에서 burn rate 오탐을 발견하고 Probe-based SLI로 전환한 판단 경험.

정의한 SLO

API Error Budget99.5% (월 에러 버짓 216분) — Request-based
API P95 Latency< 1,000ms — 부하테스트 기반 v1(500ms)→v2 조정
Chat Service99.0% — Probe-based (Blackbox Exporter)
Participation API99.0% — Request-based Ratio

실환경 발견

  • 저트래픽 서비스에 Burn Rate 적용 시 1건 에러 = 극단적 수치 → false alert 남발
  • SLO-3 probe 기반 정의했으나 실제 Blackbox가 probe하지 않는 상태 발견 → /chat-health 엔드포인트 추가
  • 부하테스트 threshold가 SLO v1 기준에 머물러 있는 것 발견 → 13개 스크립트 일괄 정렬
